# MAB Harness — Multi-Team Adversarial Review (2026-06-30)

5 independent fresh-context lenses; 33 findings adversarially verified, 12 confirmed (by severity: {'medium': 2, 'low': 6, 'none': 4}).

## Lens verdicts

### validity
The measurement validity of this harness is PARTIALLY FLAWED, with three critical confounds that prevent it from cleanly isolating memory contribution. First, the "answer from memory" instruction is applied identically to both control and memory armsâ€”it requests memory-first behavior but cannot prevent Claude from using parametric knowledge, which invalidates the control baseline. Second, production runs show no control data in reports, eliminating the only mechanism to measure memory-lift; the design intends control_arm_enabled=True but actual benchmark runs report zero paired questions. Third, the questions are predominantly answerable from parametric knowledge (historical figures, sports, geographyâ€”all in Claude's training distribution), making it impossible to distinguish memory retrieval from model knowledge overlap even if controls worked. The grading metrics (substring/exact-match) validate answer correctness only, not memory provenance. Together, these issues mean the harness measures "nous accuracy on MAB" but does NOT validly measure "memory contribution above parametric knowledge." Secondary issues include insufficient verification that recalled facts were actually injected into the Claude prompt, potential residual activation leakage from open control sessions, possible in-prompt gold leakage, and settle timeouts that silently accept incomplete consolidation. The design is sound in intent (per-instance isolation, fresh sessions, event-based settle) but the execution conflates memory with model capability and lacks controls to separate them.


### lifecycle
The harness correctly orders the memory lifecycle (control â†’ ingest â†’ consolidate â†’ answer), and ingest settling detects real completion via event-based signals (episode_summarized count increase). However, there is a critical flaw: consolidate() can timeout and return False, but the runner doesn't check or log this, allowing questions to be answered against incompletely consolidated memory. Settle timeouts are tracked in stats but not logged per-instance. The real-world nous memory chain (session_ended â†’ episode_summarized â†’ fact_extractor â†’ sleep phases) is invoked correctly, but the harness lacks protective guards around async event propagation and timeouts.

### isolation
The benchmark harness exhibits a significant isolation vulnerability within each MAB instance: control-arm and memory-arm answers run on the same nous server with the same NOUS_AGENT_ID, yet recall queries are scoped to agent_id only (not session_id). Facts learned during control-arm Q&A via LLM tool calls (learn_fact) persist to the shared eval database and contaminate memory-arm recall queries, directly inflating memory-lift measurements. Primary mitigations exist (answer instructions don't encourage learning, simple Q&A is unlikely to trigger tool use) but are insufficient and model-dependent. Secondary risk: unclosed control sessions may have episodes extracted via summarization during shutdown, persisting facts to the shared database. Code comments incorrectly justify isolation by claiming unclosed sessions can't leakâ€”this reasoning is flawed because database persistence is independent of session state. Cross-instance isolation via unique agent_ids is sound. Ingest consolidation is properly isolated. Recommendation: either use separate agent_ids for control/memory arms, add session_id filtering to recall queries, or explicitly close control sessions after control phase and implement strict LLM guardrails.

### grading
The harness has ONE HIGH-SEVERITY INTEGRITY ISSUE and several design tradeoffs. The substring_exact_match grader produces FALSE POSITIVES when agents mention gold answers while abstaining (e.g., 'I cannot find information about 43' where gold='43' is marked CORRECT). This artificially inflates baseline accuracy and critically, creates FALSE MEMORY-WINS in lift calculations where an abstaining control arm ('I don't know') is marked wrong but an abstaining memory arm ('I still can't find X') is marked correctâ€”making the memory pipeline appear to add lift when it should not. The error-handling logic and lift/accuracy pairing are otherwise correct and consistent. Word-boundary diagnostics use stricter matching than the grader itself, creating rare edge-case misattribution for numeric substrings. The test suite does not cover the abstention false-positive scenario, leaving the bug unvalidated."

### reliability
The harness has critical silent failure modes that allow invalid-but-valid-looking results. Two major issues: (1) the return values of consolidate() and ingest settle are completely ignored by the runner, allowing memory answers to be graded when consolidation/ingest failed to completeâ€”this silently invalidates the core memory-lift measurement without corrupting the data structure; (2) event endpoint unavailability causes silent degradation from event-based to timing-based settle logic, creating race conditions where answers might be given before consolidation finishes. The ingest truncation silently drops context, and control arm failures create unpaired questions. The harness reliably marks exceptions as errors but logical failures (timeout but keep going) remain invisible.

## Synthesis

## Verdict

**Sound-with-fixes, pending resolution of two unverified validity threats.** The harness exercises nous's real memory lifecycle end-to-end with no mock/shortcut path: ingest `/chat` turns â†’ DELETE session (`session_ended`) â†’ settle on `episode_summarized` + fact quiescence â†’ `/sleep/trigger` â†’ settle on `sleep_completed`, then answer from a fresh session. That chain was traced link-by-link against both harness and nous source and confirmed correct (adapter.py:176-249, runner.py:113-120, nous layer.py:1955 â†’ episode_summarizer â†’ fact_extractor â†’ sleep_handler). Lift is computed only on properly paired questions (report.py:77), and errored questions are recorded distinctly rather than zero-scored. The confirmed defects are real but mostly observability/reporting gaps that bias measurement conservatively (downward) rather than fabricating memory-lift; none corrupt the stored data structure. The harness therefore reliably measures "nous accuracy on MAB," and its lift mechanics are methodologically sound. What it has NOT been shown to measure is "memory contribution above parametric knowledge" â€” the validity and isolation lenses raise serious confounds (below) that were never put through adversarial verification, so the numbers should not yet be trusted as memory-attribution evidence until those are settled.

## Confirmed issues (ranked)

### Medium
- **Consolidate timeout is neither checked nor recorded in the report** (raised by lifecycle + reliability lenses â€” merged). `mab/runner.py:115` calls `await method.consolidate()` and discards the bool; `mab/adapter.py:240-241` logs a warning and returns `False` on a real sleep-settle timeout. Unlike ingest's `settled` flag (surfaced as `settle_timeouts`, report.py:118), the consolidate result leaves no trace in structured output, so a post-hoc analyst cannot identify or exclude instances where sleep consolidation did not settle. Verification tempered the original "silently corrupts validity" framing: facts are already durable via `_settle_ingest` before consolidate runs, `False` can be a detection false-negative from flaky `/events` polling, and a true miss biases lift *downward*. **Fix:** record the consolidate-settle status on the result (mirroring `IngestStats.settled`) and surface it in the report so affected instances can be flagged. Do **not** raise/error the instance â€” that discards valid data.

### Low
- **Substring grader marks abstentions containing the gold token as CORRECT.** `mab/grading/graders.py:48-54`: an answer like "I cannot find information about 43" with gold `["43"]` returns `correct=True`. This is the documented upstream MemoryAgentBench `substring_exact_match` semantics, and the symmetric arm framing (config.py:15-22) means abstentions are usually "I don't know" (no gold echo), so the feared asymmetric "false memory-win" is largely refuted. Residual real issue: generic leniency for short/common-token golds. **Fix:** add abstention detection (e.g. "don't know", "can't find", "no information") and reject substring matches inside abstentions, or document the leniency as accepted-by-design.
- **No test covers the abstention false-positive path.** `tests/test_graders.py` / `tests/test_lift.py:62` ("I don't know." vs `["paris"]`) never exercise an abstention that echoes gold. **Fix:** add the `("I cannot find 43", ["43"])` case and an exact-match abstention case; document expected behavior.
- **Ingest settle timeout not logged per-instance.** `adapter.py:182` / `runner.py:113-114` / `report.py:118`: aggregate `settle_timeouts` is surfaced but not broken down by instance_id/source; the warning at adapter.py:208 lacks an instance_id. Observability only. **Fix:** log instance_id/source on settle timeout; optionally emit per-instance settled flag.
- **Silent event-endpoint degradation in settle logic.** `adapter.py:147-158` returns `-1` on endpoint failure with no log; `_settle_ingest` (line 188/195) then skips the `episode_summarized` gate, and consolidate (line 234) disables the `sleep_completed` signal â€” both unlogged. Verification found the trigger improbable (the `/events` endpoints are in-memory reads, returning 0 not -1 when empty) and refuted the consolidate "race" (degraded consolidate waits *longer*, to full timeout). The one genuine correctness risk is a premature `_settle_ingest` return if the endpoint fails *and* fact-count is coincidentally stable. **Fix:** log explicitly when falling back to quiescence-based settling; track which gates were used per instance.
- **Ingest chunk truncation not visible per-instance.** `adapter.py:55-63` truncates beyond `max_ingest_chunks` (default 60, config.py:59), logging a warning (adapter.py:164-167) but `report.py:110-118` never emits `chunks_truncated`/`notes` in JSON or markdown. Dropped trailing chunks can make a never-ingested needle look like a recall failure, and write_loss attribution conflates harness truncation with genuine nous write loss. **Fix:** expose `chunks_truncated` per-instance in JSON/markdown and flag questions from truncated instances.
- **Control-arm exceptions create unpaired questions with no low-coverage warning.** `runner.py:103-110` stores control exceptions; `runner.py:141-142` set `control_correct=None`; `report.py:77` excludes them from `n_paired`. `n_paired` is surfaced (markdown lift table, JSON line 143), but there is no automatic alert when `n_paired << n_graded`, even though the report already emits an analogous coverage warning for attribution "unknown" (report.py:227-233). **Fix:** warn in markdown when paired coverage drops (e.g. < 70%).

### None / cosmetic
- **Rate-limit exhaustion reported as generic "server 500."** `adapter.py:78-107` retries nous-masked 429s then re-raises; `runner.py:124` records `"HTTPStatusError: server 500"`. No measurement impact (recorded as error, not zero-scored) and nous collapses the cause upstream so the harness cannot reliably assert `rate_limited`. Optional: log which errors were retried.

## Completeness critic

The confirmed-findings pass covered lifecycle mechanics, grading, isolation-of-ingest, and reliability/observability well. The **validity and isolation lens verdicts were never run through adversarial verification**, and they contain the threats most likely to actually invalidate the headline numbers. Before trusting any "memory-lift" figure I would settle, in order:

1. **Is there control-arm data in production runs at all?** The validity lens asserts production reports show *zero paired questions* despite `control_arm_enabled=True`. If true, lift is unmeasurable and every lift number is vacuous â€” this is potentially the single most important issue and is entirely unverified. **Check first:** open an actual production report and confirm `n_paired > 0`.
2. **Parametric-knowledge confound.** If MAB questions (historical/sports/geography) are answerable from Claude's training distribution, even working controls cannot separate memory retrieval from model knowledge. **Check:** sample the dataset and measure control-arm accuracy on memory-free questions; high control accuracy means low measurable headroom.
3. **Control-arm contamination of the shared agent_id.** The isolation lens claims control and memory arms share `NOUS_AGENT_ID` and recall is scoped to agent_id (not session_id), so any `learn_fact` tool call during control Q&A persists and inflates memory recall. **Check:** confirm whether control-arm sessions can trigger writes, and whether control runs before memory on a shared agent_id; consider separate agent_ids per arm.
4. **Did recalled facts actually enter the answer prompt?** No finding verified that the facts nous recalled were injected into Claude's context (vs. answered from parametric knowledge). The `gold_in_memory` diagnostic checks storage, not prompt-injection. **Check:** assert recalled-context presence in the answer payload.

Until items 1-3 are resolved, the harness is trustworthy as a nous *accuracy* benchmark but not yet as a memory-*attribution* benchmark.
