# nous Storage & Retrieval Root-Cause Findings (Conflict-Resolution)

**Date:** 2026-07-02
**Context:** MAB harness re-baseline of nous on Conflict-Resolution (CR), after enabling
`NOUS_DATE_LEG_ENABLED`, multi-cycle consolidation, `claude-sonnet-5` (foreground), and
the nous write-path fidelity fix (`282626c`). All numbers below are from the eval DB
`nous_mab` and the harness reports under `reports/`; embedding rank tests were run
directly against the stored vectors.

---

## TL;DR

- The CR re-baseline **doubled** (0.078 → 0.156 memory-lift), but the gain is entirely
  in **short (6k) contexts**; long contexts (32k/64k/262k) score **0**.
- Root cause of the long-context failure is a **layered write→read collapse**:
  1. **Write capture** was losing ~96% of large contexts (500-char/turn truncation) —
     **now FIXED** by nous `282626c`.
  2. But under the lossless config, **fact extraction is contaminated** (S2): the
     summarizer echoes its **own system prompt** as facts (0 content facts). **REGRESSION, open.**
  3. With no usable facts, **conflict resolution has nothing to work with**, so recall
     falls back to **chunks**, which miss on **depth, dilution, and conflict ordering**.
- **The load-bearing bug is S2.** CR is won at the *fact* layer (via supersession);
  chunk search is a blunt fallback that cannot resolve conflicts.

## 2. Storage (write-path) problems

### S1 — Hardcoded 500-char/turn transcript truncation — **FIXED (282626c)** — HIGH
Every ingest turn was truncated to 500 chars in `meta.transcript` (layer.py:462/1288),
and that truncated join (layer.py:1855) was the **sole source** for the stored transcript,
episode summary, extracted facts, and F067 chunks. Our harness sends 32,000-char turns →
~98% dropped at capture. A 1.1M-char context landed ≤42k chars / ≤8 facts.
**Fixed:** `transcript_message_max_chars` (default 500→8000; we set 40000). Verified:
transcript retention ~40k → **274k** chars.

### S2 — Fact-extraction contamination — **REGRESSION, OPEN** — HIGH
Under the lossless config (`episode_summary_max_chunks=0` over a 274k transcript), the
summarizer produces **0 content facts**; all 11 stored facts are verbatim echoes of its
**own system prompt** (`category='rule'`), e.g. *"candidate_facts use these categories:
preference, person, rule…"*, *"Rule F056 #379…"*, *"…returned as valid JSON only…"*.

**Root cause — ingest-preamble ↔ summarizer-prompt instruction collision:**
- The harness prepends **"Please remember the following information for later questions
  (part N/M):"** to every ingest chunk (adapter.py:173-174).
- At session end the summarizer LLM (episode_summarizer.py) — whose system prompt is a
  dense instruction set about summarization rules and `candidate_fact` categories
  (episode_summarizer.py:6-47, 37, 110-127) — reads the "*please remember… instructions
  (part 1/9)*" framing and **misinterprets the document as the user teaching it
  summarization rules**. It then regurgitates its own system prompt as the episode content.
- **Evidence:** episode title *"Memory Rules for Faithful Summarization Stored"*; summary
  *"The user provided a set of instructions (labeled part 1/9) defining rules for how the
  agent should summarize…"*; the echoed phrases appear in **facts (1–2×) but NOT in
  transcript/chunks (0×)** → generated by the summarizer, not captured; the phrases are
  verbatim from episode_summarizer.py:37/110/127.
- **Why lossless-triggered:** pre-fix, 500-char truncation left mostly raw content per
  turn → the summarizer summarized content (real `person`/`concept` facts, e.g. "Romano
  Prodi is a citizen of Italy"). Post-fix, the full "please remember (part N/9)" framing
  now dominates a 274k transcript across ~17 summary chunks → the misread fires
  systematically → 0 content facts.

**Fix directions:**
- **(primary, cheap) Harness:** reword the ingest preamble to frame content as *data / a
  document excerpt*, not "remember these instructions" (e.g. `"[Document part N/M]"`).
- **(secondary) nous:** harden the summarizer against instruction-like content (explicit
  user-content delimiters / a guard that the transcript is data to summarize, not
  instructions to follow); consider a non-conversational bulk-ingest path.

### S3 — Sparse extraction coverage even when clean — MEDIUM
Even pre-fix (uncontaminated), large contexts yielded only 4–8 content facts. The
`candidate_facts` caps (15 stable + 30 dated with `extraction_coverage_broadened=true`)
and per-chunk summarizer selection bound the total. Not the primary blocker, but limits
fact-layer coverage on large corpora.

### S4 — Embedding stored at 1536 dims from a 3072-dim model — LOW–MEDIUM
See §4. Quality ceiling, not a correctness bug.

## 3. Retrieval problems

Embedding rank test: for each question, embed with the run's model (`text-embedding-3-large`
@ 1536 dims) and rank all 528 stored chunks by cosine. Recall limit = 10.

| Question | Gold chunk rank / 528 | In top-10? |
|---|---|---|
| Aki Takase → Carnatic | 40 (jazz @1) | ❌ |
| Farragut → Denmark | 8 (token-only match) | ✅ |
| Shaman King → Greece | 21 | ❌ |
| Hard Times → MLK | 50 | ❌ |
| Henry of Blois → Church of Scotland | 16 | ❌ |

- **R1 — Recall depth too shallow (HIGH).** `episode_chunk_recall_limit=10` vs 528 stored
  chunks; 4/5 golds rank 16–50.
- **R2 — Chunks are vector-only, no keyword/BM25 leg (MED–HIGH)** (retrieval_pipeline.py:
  1013-1019). A chunk literally containing the rare gold token gets no lexical boost.
- **R3 — Semantic dilution (MEDIUM).** Multi-entity ~600-char chunks → top sim only
  0.28–0.40 (good retrieval is 0.5–0.8); gold indistinct from noise.
- **R4 — Conflict ordering inverted (HIGH for CR).** The original-value chunk out-ranks
  the injected-update chunk (jazz @1 vs Carnatic @40). Cosine has no supersession/recency
  awareness.
- **R5 — Fact layer unusable → no conflict resolution (HIGH).** Facts contaminated (S2) →
  `recall_deep` returns `recalled_fact_ids=0` on all 8 questions → chunk fallback only.
- **R6 — (unverified)** `rerank_by_score` gate may bury chunks below heart results unless
  flags align; subsumed by R1 (golds aren't in the top-10 by cosine regardless).
- **R7 — Truncated embeddings (S4)** depress all similarities, compounding R3.

## 4. Dimension truncation, explained

- `text-embedding-3-large` is natively **3072-dim**, but nous requests **1536** via the
  OpenAI `dimensions` parameter (`nous/brain/embeddings.py:44,156,195`), so the API returns
  a **Matryoshka-truncated** 1536-dim vector. (Matryoshka Representation Learning packs the
  most important signal into the leading dimensions, so a truncated prefix degrades
  gracefully rather than breaking.)
- **Why 1536:** the eval DB columns are `vector(1536)` (provisioned for the 1536-native
  small/ada default) and `embedding_dimensions` defaults to 1536.
- **Correctness:** store-time and query-time both use large@1536, so retrieval is
  dimensionally consistent and correct — **not** a broken-retrieval bug. (Verified: query
  embedded with large@1536 → top sim 0.285; small@1536 → 0.057; native large@3072 →
  dimension-mismatch error.)
- **Cost:** large@1536 ≈ small@1536, but **worse than large@3072** — the big model is run
  at half its native width, contributing to the low ~0.3 similarities (the rest is R3).
  Full quality needs `vector(3072)` columns + `embedding_dimensions=3072` + re-embed (a
  schema migration, not a config flip). **Secondary contributor.**
- **Config note:** `NOUS_EMBEDDING_MODEL=text-embedding-3-large` **is** honored (the model
  is large), just pinned to 1536 dims — earlier "config inert" phrasing was incorrect.

## 5. End-to-end causal chain

```
write fix (S1) → full context captured in chunks ✅
  BUT lossless config → summarizer misreads ingest preamble as instructions (S2)
    → fact extraction echoes system prompt → 0 content facts
      → conflict resolution (a fact-layer operation) has nothing to work with (R5)
        → recall falls back to chunks, which:
            • miss on depth (gold @16–50, limit 10)        (R1)
            • lack a keyword leg for the gold token          (R2)
            • dilute (top sim ~0.3, gold indistinct)         (R3, R7)
            • surface the ORIGINAL value over the UPDATE     (R4)
              → wrong / abstaining answers (0/8 on large contexts)
```

---

## 6. Prioritized recommendations

1. **Fix S2 (highest value).** Reword the harness ingest preamble so the summarizer does
   not misread bulk content as instructions; and/or harden the nous summarizer. This
   restores content facts and unlocks the *correct* CR path (supersession at the fact
   layer). Re-measure after.
2. **R2 — add a keyword/BM25 leg to chunk retrieval.** Cheapest retrieval win; directly
   finds chunks containing the rare gold token.
3. **R1 — raise `episode_chunk_recall_limit`** 10 → ~30–50. Recovers golds at rank 16–50
   (partial; does not fix conflict ordering).
4. **R4/R3 — longer-term:** recency/supersession-aware chunk ranking; smaller, more
   focused chunks to reduce dilution.
5. **S4 — optional quality upgrade:** migrate to `vector(3072)` + full large embeddings.

**Do not** raise `graph_backfill_max_*` (200→1000): verified non-binding — max 8 facts /
2 episodes / 41 chunks per agent, orders of magnitude below the caps.
